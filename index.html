Week 13: Rebuilding in React - Understanding "State"

Published on: November 30, 2025

1. Key Learning Moments

This week, I learned the most important concept in React: useState. In my old "Vanilla JavaScript" projects, if I wanted to change a number on the screen, I had to find the HTML element and change its text manually. In React, I just change the data (the state), and React automatically updates the screen for me. It feels like magic.

2. Personal Connections

I tried to rebuild a tiny part of my Focus Clock app—just the "Start" button—using React. It was shocking how much less code I needed. I didn't have to write document.getElementById or addEventListener even once. This makes me excited because it means I can focus more on the logic of my app and less on the boring setup code.

3. Challenges and Growth

My biggest challenge is "unlearning" my old habits. I kept trying to change things directly, but React doesn't like that. I have to trust React to do the work. It is hard to change how I think after spending 12 weeks doing it the other way. I practiced by making a simple counter button that goes up by 1 every time I click it, just to get used to the new syntax.

4. AI Tool Documentation

I used ChatGPT to help me "translate" my code. I pasted a function from my old project and asked, "How would I write this click handler in React?" Seeing the two versions side-by-side helped me understand the pattern. It showed me exactly where the useState hook fits in.

Week 12: Starting a New Challenge - My First Look at React

Published on: November 23, 2025

1. Key Learning Moments

This week, I learned that there is a very different way to build websites. I started learning a JavaScript library called React. The main idea is "components." Instead of one big HTML file, you build small, custom pieces (like a button or a timer) and put them together. It feels like building with digital LEGOs.

2. Personal Connections

I feel like I am a beginner all over again, which is both scary and exciting. As I was learning, I kept thinking, "Wow, this would have made building my timer app much easier." I can see how React helps keep big projects organized. It feels like the next big step in my journey to becoming a developer.

3. Challenges and Growth

The biggest challenge was just getting started. I had to install new tools (like Node.js) and run commands in the terminal, which was confusing. I also had to learn JSX, which looks like HTML but is actually JavaScript. My brain had to work hard to understand this new way of thinking, but by the end of the week, I built a simple "Hello, world!" component.

4. AI Tool Documentation

The official React documents were confusing. I used Gemini to help me understand the main idea. I asked, "Explain React to me like I know JavaScript but nothing else." It gave me a great simple answer, comparing React "state" to a person's memory, which really helped the concept click in my head.

Week 10: Making Changes Based on User Feedback

Published on: November 9, 2025

1. Key Learning Moments

This week, I learned how to build a settings menu. The feedback from last week showed that my app needed to be more flexible. I added a small "settings" icon that opens a panel. Now, users can turn the random reminder sound on or off. This was a great lesson in how to give control to the user.

2. Personal Connections

It felt very satisfying to fix the problems my friends found. When I made the "annoying" sound optional, I felt like I was truly improving the app, not just for me, but for others. I also added a "Start" button to the rest timer, as one friend was confused if the rest period started automatically. This made the app much clearer.

3. Challenges and Growth

My challenge was saving the user's settings. When I turned the sound off and reloaded the page, the setting was gone. To fix this, I had to learn about `localStorage`. This is a way for a browser to remember small pieces of information. It was my first time working with it, and it was tricky, but now the app remembers if you want the sound on or off, which is a big improvement.

4. AI Tool Documentation

I used ChatGPT to learn `localStorage`. I asked, "How do I use localStorage to save a 'true/false' setting in JavaScript?" It gave me two small code examples, one for saving the setting (`setItem`) and one for reading it when the page loads (`getItem`). This clear, simple example was exactly what I needed to implement the feature quickly.

Week 9: Sharing My App and Learning from Feedback

Published on: November 2, 2025

1. Key Learning Moments

I learned that building an app is only half the job. The other half is watching someone else use it. The most important lesson was: I am not my user. Things that were obvious to me (because I built the app) were confusing to my friends. I learned that feedback is a gift, even when it’s hard to hear.

2. Personal Connections

I was very nervous to share the link to my app. I sent it to a few friends and watched them use it. It was an amazing feeling to see my project work on someone else's screen! But it was also strange. They tried to click things that weren't buttons and were confused about why the random sound played. It was fascinating to see my app through their eyes.

3. Challenges and Growth

The biggest challenge was hearing negative feedback. One friend said the random sound was "annoying and stressful," not "focusing." At first, I felt defensive and wanted to explain why they were wrong. My growth this week came from learning to stay quiet and just ask, "Can you tell me more about that?" This feedback is valuable, and I realized I should make the sound an optional feature that users can turn on or off.

4. AI Tool Documentation

After talking to three friends, my notebook was full of messy notes. I used Gemini to help me organize them. I pasted all my notes and used the prompt: "Here is all the user feedback for my app. Summarize the top 3 problems and top 2 things people liked." This AI tool helped me see the most important patterns and create a clear list of what I need to fix next.

Week 8: Refactoring and Improving the User Experience

Published on: October 26, 2025

1. Key Learning Moments

This week, I learned that "working code" is not always "good code." I learned about **refactoring**, which means rewriting your code to make it cleaner and easier to understand, without changing what it does. I had one very long function that handled everything. I broke it down into several smaller functions, each with one specific job. This makes the code much more organized.

2. Personal Connections

Cleaning up my code made me feel more like a professional developer. It's not just about making something work; it's about building it well. I feel proud of my project now, not just because it works, but because the code behind it is neat and tidy. If I look at this code in six months, I will still be able to understand it easily.

3. Challenges and Growth

At first, I was scared to change my code because it was already working, and I was afraid I would break it. My solution was to change only one small thing at a time and test it carefully. I also added small user experience (UX) improvements, like making the "Start" button change color and become unclickable after the timer begins. This gives the user clear feedback and makes the app feel more polished.

4. AI Tool Documentation

I used GitHub Copilot Chat for refactoring. I copied my big, messy function into the chat and asked, "Can you help me refactor this JavaScript function to be cleaner?" Copilot suggested breaking it into three smaller functions and even gave me names for them. It helped me see a better way to structure my logic, acting like a senior developer giving me advice.

Week 7: Adding Key Features - Sounds and Rests

Published on: October 19, 2025

1. Key Learning Moments

The most important lesson this week was managing the app's "state." I learned that my app can be in different modes: a "focus mode," a "rest mode," or an "idle mode." I had to write code that always knows which state the app is in. I also learned how to use `Math.random()` in JavaScript to make the reminder sounds play at unpredictable times during the focus session.

2. Personal Connections

Adding the random reminder sounds was exciting because it's the unique feature that makes my app different from other timers. When the first sound played, the app finally felt complete and matched the original idea in my head. Now that the 20-minute rest timer also works, it feels like a real tool I can use every day to improve my own focus.

3. Challenges and Growth

My biggest challenge was a bug with the timers. Sometimes, a random sound would try to play right after the main 90-minute timer had finished. This caused a small glitch. I had to write extra checks in my code to make sure the reminder sounds only play during the "focus mode." This taught me to think about these "edge cases" (small, unexpected problems) and to write stronger code to prevent them.

4. AI Tool Documentation

To figure out how to play sounds, I used Perplexity AI. I asked a simple question: "How to play a sound file in JavaScript with one function call?" It gave me a clear code snippet showing how to create an `Audio` object and use the `.play()` method. This was very helpful and saved me from having to read long articles.

Week 6: Making it Work - Adding the Timer Logic with JavaScript

Published on: October 12, 2025

1. Key Learning Moments

My "aha!" moment this week was understanding the `setInterval` function in JavaScript. I never knew how websites updated things every second, and `setInterval` is the key. It lets me run my "update time" function every 1000 milliseconds (1 second). I also learned the math to change total seconds into a "minutes:seconds" format, which was a fun logic puzzle.

2. Personal Connections

I felt so excited when I clicked the "Start" button and the numbers on the clock began to count down. In that moment, my project changed from a static page into a real, working tool. This is the core function of my app idea, and building it myself made me feel closer to my goal of becoming a developer who can create useful things.

3. Challenges and Growth

My biggest problem was a small formatting bug. When the timer's seconds dropped from 10 to 9, my clock showed "15:9" instead of "15:09". It looked unprofessional. To fix it, I learned how to add a "leading zero" if the number is less than 10. This challenge taught me that details are very important and that I can find solutions to problems online.

4. AI Tool Documentation

I used Google's Gemini a lot to help me fix my timer code. I pasted my broken function and asked, "Why is my timer's format wrong?" Gemini quickly found the "leading zero" problem, gave me the correct code, and clearly explained why it worked better. It was like having a smart helper who could find my mistakes and teach me at the same time.

This week, I learned how HTML and CSS work together. HTML builds the page's structure, like a skeleton. CSS adds the style, like clothes. A key moment was learning CSS Flexbox. Week 4: From Idea to Concept - Designing a Focus Clock App

Published on: September 21, 2025

1. Key Learning Moments

This week, my learning focus shifted from mastering specific coding techniques to a broader 'product mindset.' The key concept that resonated with me is that a good application isn't just a collection of code, but a thoughtful solution born from a deep understanding of user needs. I spent a lot of time thinking about why people get distracted, and how a simple timer with a core feature of 'random reminders' could genuinely help users return to a state of focus. This process made me realize that clearly defining the 'problem' and the 'solution' is crucial before writing the first line of code.

2. Personal Connections

The idea for this app stems directly from my own struggles. While learning to code, I often find myself unconsciously picking up my phone or opening unrelated tabs, which hurts my productivity. Traditional Pomodoro timers sometimes feel too rigid for me. I wanted to create a tool that provides a long, immersive work block (90 minutes, based on ultradian rhythm theories) while also gently pulling me back on track in an unpredictable way. This is more than just a programming exercise; it's an attempt to build a solution for myself and for others who face the same challenge.

3. Challenges and Growth

The biggest challenge wasn't technical, but conceptual: how to design the 'random sound' feature so that it serves as a helpful reminder instead of becoming a new source of distraction. I considered different sound types (like a soft chime or a water drop) and the random time intervals for the alerts (should it be completely random, or random within a specific range?). To work through this, I sketched out a User Flow Diagram, simulating the entire user experience from starting the timer to finishing the rest period. This process forced me to think about every detail from the user's perspective, marking a small but significant step for me from being just a 'code implementer' to becoming a 'product designer.'

4. AI Tool Documentation

During the ideation phase, I used Miro (an online whiteboard tool with AI features) for brainstorming. I created a mind map to organize the app's core features, target users, and potential challenges. When I got stuck, I used Miro AI's 'expand idea' feature. I typed in the prompt 'how to make alerts non-intrusive,' and it generated several keywords for me, such as 'ambient sounds,' 'adjustable frequency,' and 'subtle visual cues.' These suggestions significantly broadened my thinking and helped me design the app's core experience more holistically.

Week 3: Putting Theory into Practice - Building My First Interactive Component

Published on: September 7, 2025

1. Key Learning Moments

This week, the focus shifted entirely from theory to practice. The activity that resonated most was building a simple to-do list application. This process gave me a deep understanding of how different concepts connect: HTML for structure, CSS for style, and JavaScript as the glue that brings it to life. For the first time, I clearly felt the power of DOM manipulation—dynamically creating, reading, updating, and deleting (CRUD) elements on the page with JavaScript. It was more exciting than any isolated exercise before it.

2. Personal Connections

This hands-on project connects directly to my long-term goal of becoming a front-end developer. It was no longer an abstract code snippet but a tangible 'product' I could see and interact with. This small success significantly boosted my confidence and made me believe in my ability to turn ideas into reality. It also got me thinking about my personal portfolio; while this to-do list is simple, it serves as a cornerstone for the more complex applications I plan to build.

3. Challenges and Growth

The biggest challenge appeared when handling user input and event listeners. Initially, my code worked for adding new tasks, but when I tried to delete a specific one, it would either delete the wrong item or not react at all. I spent an entire afternoon using `console.log()` to trace the values of variables and the execution flow of my functions step-by-step. I eventually discovered a logical error in how I was getting the index of the clicked element. While the process of fixing this bug was frustrating, it taught me how to debug systematically and showed me the importance of using the browser's developer tools. This was where I experienced the most growth.

4. AI Tool Documentation

I used GitHub Copilot to assist with coding during this challenge. When I got stuck on a tricky JavaScript function, instead of asking it to write the entire code, I first wrote my goal in a comment (e.g., `// function to find and remove an item from the todos array by its id`). Copilot then generated a very clean and efficient code suggestion. I carefully studied its suggested code and compared it to my original approach. This not only solved my problem but also taught me a more optimized way to write it. In this context, it acted like a senior pair programmer.

Week 2: The Nuances of AI

Published on: August 31, 2025

1. Key Learning Moments

This week, the concept of 'asynchronous operations' in JavaScript finally clicked. The discussion around the event loop and how callbacks, promises, and `async/await` all interact was a major breakthrough. The activity where we had to refactor a callback-heavy function to use `async/await` was particularly insightful; it demonstrated the dramatic improvement in readability and maintainability.

2. Personal Connections

This connects directly to my goal of building more responsive and efficient web applications. In past projects, I've run into issues with slow, blocking operations that froze the user interface. Understanding asynchronicity gives me the tools to prevent that, which will be critical for the personal portfolio project I'm planning.

3. Challenges and Growth

My main challenge was debugging a promise chain that wasn't resolving in the order I expected. I spent a couple of hours trying to console.log my way through it. I worked through this by drawing out the execution flow on a whiteboard, step-by-step. This visual approach helped me identify a `.then()` block that was missing a `return` statement, which broke the chain. It was a frustrating but valuable lesson in careful debugging.

4. AI Tool Documentation

I used **Google's Gemini** this week to enhance my reflection. After struggling to articulate my challenge, I prompted Gemini: "Explain the common pitfalls of JavaScript promise chains in simple terms." Its clear explanation, which included the 'missing return' issue, helped me structure my thoughts and write the 'Challenges and Growth' section with more clarity and technical accuracy.

Week 1: Foundations of AI

Published on: August 24, 2025

1. Key Learning Moments

The most resonant concept this week was the fundamental difference between 'declarative' and 'imperative' programming paradigms. Seeing the same problem solved both ways highlighted how a declarative approach (like in HTML or SQL) focuses on the 'what', while an imperative approach (like in vanilla C) focuses on the 'how'. The in-class discussion comparing React components (declarative) to jQuery DOM manipulation (imperative) was a real lightbulb moment.

2. Personal Connections

This relates to my interest in user interface design. A declarative mindset seems much more aligned with designing components and states, rather than manually writing step-by-step instructions to change every little thing on the screen. It feels like a more organized and scalable way to think about building complex UIs.

3. Challenges and Growth

I found it difficult at first to grasp the scope of variables in our initial coding exercises. I kept declaring variables inside loops, expecting them to be accessible outside, which led to several errors. I worked through it by reading the official documentation and deliberately practicing with small, isolated code snippets until the rules of block-level vs. function-level scope became second nature.

4. AI Tool Documentation

I used **ChatGPT** to help brainstorm analogies for the declarative vs. imperative concept. I prompted it to "Provide five real-world analogies to explain declarative vs. imperative programming." It gave me several useful examples (like ordering a pizza vs. giving the chef a step-by-step recipe), which helped solidify my understanding and gave me a great way to explain it in my notes.
